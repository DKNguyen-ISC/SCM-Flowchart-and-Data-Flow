This documentation explains the core logic, strategies, and rationale behind the data flow model.

1. Scope & Purpose

This model visualizes the "To-Be" (Phase 2) Procure-to-Pay (P2P) process. It is designed to track material requirements from planning through to payment, emphasizing data integrity, audibility, and the separation of transactional processes from master data.

2. Core Logic: The "Transformation Pattern"

A standardized pattern is used for any process involving offline user work (like Excel). This ensures data is validated before corrupting final tables.

  • Draft Document: A user's initial request (e.g., `Bom_Order_List_Draft`).
  • Offline Event: The user downloads the draft, enriches it offline (e.g., `Select_BOM_Update`).
  • Staging Table: The enriched file is uploaded to a temporary table (e.g., `Bom_Order_List_Staging`).
  • Validation: The system runs automated checks on the staging table (e.g., "Does this supplier exist?").
  • Final Document: If valid, data is moved to the permanent, auditable table (e.g., `Bom_Order_List_Final`).

3. Key Architectural Modifiers

  • [H] Tracks History (SCD2): Applied to master data (`Supplier_Information`) and key auditable documents (`Purchase_Order`). It means we never delete or overwrite records; we expire old ones and create new ones.
  • [A] Aggregated / Calculated: This is not a table, but a *view* or *calculation*. `Stock_Data` is an aggregated sum of all movements in the `Stock_Ledger`.

4. Module-Specific Rationale

  • M1. Sourcing in Planning: Sourcing (Supplier & Price selection) is intentionally done during the `Select_BOM_Update` (Planning) step. This locks in the supplier and price *before* the Purchase Request is even created, streamlining M2.
  • M2. Decoupled Procurement: M2 is a chain of these transformation patterns. This decouples the "Net Demand" (from M1) from the final "Purchase Order," allowing for PR consolidation and review.
  • M4. Perpetual Inventory: The `Stock_Ledger` is the single source of truth for all quantity changes. Every event (Receipt, Issue, Return, Adjustment) *must* post a transaction to the ledger.
