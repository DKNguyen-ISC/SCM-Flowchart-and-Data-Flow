MODULE 3: SUPPLY & DEMAND BALANCING

Process Overview (v12.0)

This is the **"Brain"** of the system. This hub consumes Gross Demand (from M1), On-Hand Stock (from L1), and Purchase Orders (from M2) to calculate the true Net Demand.
Key Logic: It uses a **"Soft Pegging"** model. The `Manage_Pegging` workbench allows a PIC to dynamically re-allocate "OnTheWay" supply to meet emergency demands, and the system immediately calculates the consequences.

1. Balancing & Pegging (Initial State)

The system runs its initial calculation. The **Balancing (left)** table calculates the plan. The **Pegging (right)** table is the "brain" that tells the calculation which POs are "soft-pegged" to which VPOs. Initially, all POs from M2 are pegged 1-to-1.

2. Manage_Pegging (The "Workbench" Event)

This is the "human-in-the-loop" action. An emergency order (`VPO-C05`) for **Dec 15** arrives. The PIC decides to "steal" the supply from `PO-101` (originally for `VPO-C04` on Dec 14) to cover this new, more urgent order.

Emergency: New `VPO-C05` (Req: Dec 15) needs **14,040** units of `300010406C`.

Action 1: PIC *breaks* the peg: `PO-101, L1` (14,040 units) -> `VPO-C04`.

Action 2 (New Peg): PIC *re-pegs*: `PO-101, L1` (14,040 units) -> `VPO-C05`.

This action updates the Pegging table...

3. Balancing & Pegging (Updated State)

The system re-calculates the entire plan based on the PIC's change.
Right Table: The Pegging table now shows `PO-101` is linked to `VPO-C05`.
Left Table: The Balancing calculation updates. `VPO-C05` (Dec 15) is now Covered, but `VPO-C04` (Dec 14) is now Uncovered.

4. Final Output to M2 (Purchase_Request_Draft)

This is the **final output of Module 3**. The system filters the updated balancing table to show *only* the lines with a `NET_DEMAND > 0`. This is the true "To-Do List" that is fed back to **Module 2: Procurement**.

Notice that `VPO-C04` (the order we "stole" from) is now back on the "to-buy" list.
